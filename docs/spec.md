# hakoniwa-aquarium

## 概要

ブラウザで遊ぶドット絵の箱庭ゲーム。不思議な生物が棲む「海域」が舞台。主人公の子ネッシーがリング状の海を泳ぎ、影の魚（シルエットの餌）を食べると住民が生まれ、図鑑（この海域の記録）が育つ。お菓子要素は前面に出さず、フレーバー（配色）として静かに存在させる。

## 解決する課題

1. **オーナー自身が、すきま時間に遊んで癒される箱庭が欲しい。** 勝敗・スコア・時間制限のあるゲームは休憩にならない。眺めて・なぞって・少しずつ増えていく、負荷の低い遊びが必要
2. **既存の放置ゲー・箱庭アプリは課金導線や通知が多く、静かに遊べない。** 自作なら世界観・ペース・操作感をすべて自分好みに調整できる
3. **導入前**: 休憩時間にSNSを眺めて疲れる / **導入後**: 自分の海域を眺め、餌をあげ、図鑑が1体ずつ埋まっていく楽しみが積み重なる
4. 放置すると「自分のための小さな楽しみ」を持たないまま休憩の質が下がり続ける。図鑑が育つ長期の楽しみは既製品では代替しにくい

## ゲームコンセプト

- **不思議な生物が棲む海域**: 影の魚を食べると「未確認」だった生物が姿を現し、住民として泳ぎはじめる。図鑑はこの海域の観測記録
- **図鑑が育つ**: 誕生・目撃で図鑑が埋まっていくことが長期の駆動力。コンプリート体験を壊さないよう、誕生順は固定テーブルで管理する（ランダム禁止）
- **お菓子はフレーバー**: 生物のお菓子性は配色でのみ表現し、形状は実在種を構造テンプレートとして借りる（詳細はアートディレクション規則）

## 機能一覧

- 線描き操作による主人公の移動（なぞった軌跡を道のりとして消費）
- タップ1回＝1点の道のり（線描きの下位互換として機能）
- 影の魚の捕食（頭の位置でのみ判定）
- 満腹5で住民1体誕生（固定テーブル順）
- 誕生した住民が海域を泳ぐ
- 深度による背景グラデーション（浅瀬→中層→深海＋海底の砂）

## 操作仕様

- 画面をなぞる → 4px間隔サンプリング・最大60点のウェイポイント列を生成し、先頭から消費する
- タップ → 1点のウェイポイントとして同じ仕組みで処理する
- 具体的な定数（サンプリング間隔・最大点数・到達判定距離・入力クランプ）は `src/data/world-constants.ts` を正とする

## エンジン仕様

数値の正は `src/data/world-constants.ts`。ここには構造と設計原則（なぜそうなっているか）を書く。

### torus 世界

- ワールドは W=480, H=160 のリング状。x座標はすべて torus（`mod(x, W)`）で扱い、**端の分岐コードを書かない**
- torus距離: `d = mod(a-b, W); if (d > W/2) d -= W;`
- ビューポートは 128×90（論理解像度）。CSSで整数倍拡大し `image-rendering: pixelated`
- カメラは主人公中央追従。xはtorus、yは `clamp(y - VH/2, 0, H-VH)`

### 物理（推進・水流・抗力）

- 固定タイムステップ（60fps相当のtick）。物理定数はすべて「1 tickあたり」の値。タブ復帰時の暴走は1フレームあたりのtick消化上限で防ぐ
- 推進は「常時の弱い推進＋周期的なひと掻きバースト」の2層。掻きフレームとスプライトアニメを同期させる
- 水流は常時 +x 方向。**設計原則: 水流の終端速度 < 主人公の持続推進**。これを破ると左移動が不能になり、リング世界が構造的に壊れる。難易度としての増減はこの原則の範囲内で自由
- 向きは vx の符号で即時鏡像反転（レトロ文法）

### 捕食判定

- **頭の位置**（向きに応じたオフセット）と餌の距離で判定する。胴では食べられない（意図した難易度）
- 餌（影の魚）は同時3匹・一定方向に周回。食べられたら反対側にリスポーン

### 待ち伏せ型の設計原則

ゲーム構造は**待ち伏せ型**。餌は必ず全xを通過するので、詰みが構造上存在しない。よって水流・餌速度は自由に難易度調整に使ってよい。プレイヤーの主な意思決定軸は「どの深度で張るか」。

## サイズ階級（確定）

枠は上限。枠内の占有率は生物の格の表現に使ってよい。1ドットの表示スケールは全階級共通（mixels禁止）。輪郭は全階級1px。隣接階級の線寸比は1.5前後以上を維持する（1.33では区別できないことを実測済み）。

| 階級 | 枠 | 役割・例 | 制作ルール |
|---|---|---|---|
| SS | 〜8×8 | 影の魚（餌） | 1色シルエット・アニメ省略可 |
| S | 〜16×16 | ラムネ魚・クラゲ・クマノミ・たい焼き | 5色ランプ+目・2フレーム |
| M | 〜24×20 | 子ネッシー（主人公） | 5〜7色・2フレーム |
| L | 〜40×24 | マンボウ・ウミガメ・イルカ級 | 7色可・登場演出込み |
| LL | 〜64×32 | ジンベエザメ・クジラ（回遊レア） | 同上 |

## アートディレクション規則（確定）

1. **生物のお菓子性はフレーバー（配色）で表現する。モチーフ（形状）では表現しない**
2. 形状は実在種のパターンを構造テンプレートとして借りる（可読性と生態記号を保つ）
3. 形状モチーフのお菓子はプロップ（地形）側が担当する（ウエハース岩・金平糖・たい焼き屋台等）
4. 例外: たい焼きは「お菓子自身が魚の形」なので生物側で許可。剛体（変形フレーム禁止・動きは全体ボブのみ。振幅は species-motion.ts を正とする）
5. 量産式: 形状テンプレート × フレーバーパレット = 1住民。パレット差し替え時はベース色だけでなく全ランプを同一色相で回す
6. selout（色相を保った暗色輪郭）。孤立1pxドットはノイズ化するので禁止（模様は2px以上か、1px幅でも縦横に連続させる）
7. 完全に想像上の生物であることを宣言するため、実在種にありえない配色を積極的に使ってよい（例: いちごミルク・クマノミ）
8. エフェクトも1ドット＝論理1px。論理キャンバスへのベクター描画（arc/ellipse等）は禁止（mixels回避）

スプライトの実データ（グリッド文字列・パレット）は `src/data/sprites/` を正とする。パレットは種ごとにネームスペース分離する（グローバル共有パレット禁止）。

## 滞在スケジューラ（M2以降・確定仕様）

- 3クラス: **常駐**（滞在∞。特定プロップに紐づく生物。例: サンゴ+クマノミ）/ **通い**（誕生直後はしばらく確実に滞在、以後は来訪抽選）/ **回遊**（出現重み極小・画面を一横断して去る。LL級）
- 顔ぶれはセッション開始時に図鑑から抽選で生成。**実時間減衰は禁止**（数日ぶりに開いて過疎、を防ぐ）。時間経過はプレイ中のみ
- 退場は**カメラ外でのみ**発動する（自分の魚が消える瞬間を見せない）
- 退場演出: パレットを背景の水色へlerpして影に戻ってから消す（遠のく=再び未確認になる、の世界観と一致）
- 寂しさ防止の床: 常駐+最低来訪数をセッション抽選で保証する
- 表示予算: 同時表示数を定数で制御する（餌n匹+住民m体）
- セーブは「図鑑の発見状態+常駐」のみ。泳いでいる顔ぶれは保存しない

## 誕生メカニクス（M2以降を含む確定仕様）

- 影の魚を5匹食べる→住民1体誕生。誕生順は**固定テーブル**（`src/data/birth-table.ts`。ランダム禁止。図鑑コンプ体験を守る）
- 誕生ペーシング: 1日あたりの誕生数上限などで図鑑の寿命を制御する
- 拡張予約: 常駐の解禁条件＝特定プロップ設置のフラグ（例: サンゴ設置→クマノミ解禁、たい焼き屋台→たい焼き常駐化）
- 回遊レアは「誕生」でなく「目撃」として図鑑記録。初回目撃は影で載り、複数回目撃でフルカラー解禁

## 意図的な仕様（決定ログ）

バグと区別がつきにくい意図的な仕様と、その理由を記録する。新たな決定はここに追記する（重量級のADRは作らない）。

1. **セレモニー中に引いた線は保持し、演出明けに出発する**
   - 決定: 誕生セレモニー中に引いた線はクリアせず保持し、演出が明けたらその線に沿って出発する
   - 理由: 演出中も描き直しで上書きできるため「最新の意思が常に勝つ」一貫性がある
   - 再検討のトリガー: 演出明けの急発進が興ざめに感じられたら誕生時クリア案
2. **同tickに複数捕食した場合、満腹の超過分は次の誕生へ繰り越す**
   - 決定: satiety 4 で同tickに2匹食べたら誕生1回・残り1（`satiety -= SATIETY_MAX` で繰り越す）
   - 理由: 切り捨ては1食分が消え理不尽（2026-07-05 オーナー指摘により再検討トリガー発動、切り捨てから変更）
   - 安全性: BAIT_COUNT=3 より1tickの最大加算は3 → 繰り越しは最大2で二重誕生は構造的に不可能。セレモニー中は捕食無効なので連鎖もなし
   - 再検討のトリガー: BAIT_COUNT や SATIETY_MAX の変更で「繰り越し ≥ SATIETY_MAX」があり得るようになったら
3. **住民が満員（RESIDENT_MAX=8）でも誕生する（押し出し方式）**
   - 決定: 満員でも誕生する（押し出し方式）。新生児と同じサイズ階級の住民（新生児除く）からランダムに1体が退場予定となりカメラ外で消える。一時的に9体になる期間を許容する
   - 押し出し対象の選定: 同サイズ階級の住民がいなければ新生児以外の全住民からランダム（フォールバック）。最古参ルールは不採用（長くいた子が消えるのは寂しい・大型レアが小魚の誕生で押し出されるのを防ぐ）
   - 旧決定: ~~満員時も `satiety -= SATIETY_MAX` で消化する（超過分の繰り越しも決定2と同じ）。誕生だけを抑制する~~
   - 変更経緯: セッションスケジューラ導入に伴い「食べたのに生まれない」体験を解消（2026-07-07）。旧決定の再検討トリガー「スケジューラ実装で満員の意味が変わったら」が発動。設計: docs/design/2026-07-07-session-roster.md
   - 再検討のトリガー: 押し出しが頻発して顔ぶれが落ち着かないと感じたら（RESIDENT_MAX や誕生ペーシングで調整）
4. **誕生セレモニー中は捕食判定を無効にする**
   - 決定: セレモニー中は餌に触れても捕食しない（構造的に satiety=0 のまま固定される）
   - 理由: 誕生の連鎖で演出が重なるのを防ぐ
   - 再検討のトリガー: セレモニー短縮や演出の並行許可を検討するとき

## データ永続化

- localStorage による自動セーブ実装済み（2026-07-06。設計: docs/design/2026-07-06-zukan-save.md）
- 保存するもの: version（スキーマ番号）+ 図鑑（種→発見日・誕生数）+ 満腹ゲージ
- 保存しないもの: 泳いでいる顔ぶれ・位置（顔ぶれは起動時に図鑑から抽選で生成する）
- タイミング: 誕生・捕食のたびに自動保存。手動セーブなし
- 常駐の保存は滞在スケジューラ実装時に追加予定

## 技術構成

- 言語: TypeScript
- ランタイム: Bun（パッケージ管理・テストランナー）
- ビルド: Vite + 素の Canvas 2D（フレームワーク・ゲームエンジン不使用）
- lint/format: Biome
- CI/CD: GitHub Actions（lint / tsc / test / build）→ Cloudflare Pages デプロイ（wrangler-action）
- 一関数一ファイルを厳守。純粋関数（torus距離・深度lerp・誕生テーブル等）はすべて bun test でテストし、描画はテストしない

## テスト観点

- 正常系: torus距離・カメラ変換・ウェイポイント消費・捕食判定・満腹→誕生の遷移が仕様どおり動く
- 異常系: 満員時の誕生抑制 / 入力座標のクランプ / タブ復帰時のtick消化上限
- エッジケース: torus境界（x=0/W付近）をまたぐ距離・捕食・描画 / ウェイポイント最大数到達時の挙動
